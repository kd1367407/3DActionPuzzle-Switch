# 【Switch】

## 作品概要
・ジャンル：3Dアクションパズルゲーム
・制作人数：個人制作
・開発環境：C++, 学校提供のライブラリ
・概要：ブロックの配置を入れ替えながら進むパズルアクション

## ソースコードの見どころ

本リポジトリはゲーム全体のソースコードを含んでいますが、ゲームエンジン開発・パイプライン構築の視点から、特に注力して設計した基盤システム群を以下にまとめました。

### 1. 空間分割（グリッド分割）と衝突判定の最適化
▶ `Application/System/PhysicsSystem.cpp` ( `AssignGrid`, `Update` 関数周辺)
総当たりによる計算負荷（O(N^2)）を削減するため、空間を一定サイズのセルに分割する一様グリッド（Uniform Grid）を実装しました。
生存しているコライダーのみを毎フレーム登録し直し、同一・隣接セル内のオブジェクト間でのみ詳細なメッシュ判定（Narrow-phase）を行うことで、大量のオブジェクトが存在するステージでもパフォーマンスを維持しています。
**【今後の課題】**
現在の重複判定（m_checkedPairs）でポインタアドレスから文字列を生成してキーにしている部分は、毎フレームの動的なメモリ確保（std::stringの生成）が発生し、パフォーマンスのボトルネックになり得ると認識しています。
そのため今後は文字列変換をやめ、**「2つのオブジェクトID（32bit）をビットシフトで結合し、1つの64bit整数（uint64_t）として直接キーに利用する」**という、軽量でメモリアロケーションを伴わないペアキャッシュ機構へのリファクタリングを想定しています。

### 2. 可変フレームレートに対応した物理挙動（サブステッピングの実装）
▶ `Application/Scene/Basescene/BaseScene.cpp` ( `PostUpdate` 関数周辺 )
PCのスペックによるFPSの変動が、プレイヤーのジャンプの高さや挙動のブレに影響を与えないよう、物理演算にサブステッピングを導入しました。
`deltaTime`を理想フレーム（1/60秒）で分割して小刻みに`PhysicsSystem::Instance().Update()`を回すことで、すり抜けの防止と挙動の完全な安定化を実現しています。（※処理落ち時のフリーズ対策として上限回数も設けています）

### 3. データ駆動設計によるレベル構築（コンポーネント指向）
▶ `Application/GameViewModel.cpp` ( `PopulateSceneFromEntities` 関数周辺 )
▶ `sample_stage.json` (※実際のjsonファイルパスに合わせてください)
ステージ上の全オブジェクトをハードコードせず、JSONファイルから動的に構築するデータ駆動のパイプラインを作成しました。
JSONの構造はECS（Entity Component System）を意識しており、`Archetype`（型）に対して必要な`Components`（Collider, Dissolve, Transformなど）のパラメータを注入する仕組みにすることで、プログラマーの手を借りずにプランナーやAIが自由にステージを生成・拡張できる基盤を作りました。

### 4. MVVMアーキテクチャとCommandパターンの導入（エディタ機能）
▶ `GameViewModel.cpp` 
▶ `StageModel.cpp`
レベルエディタの操作やゲームの進行管理において、UI（View）とデータ（Model）を疎結合にするためMVVMパターンを採用しました。
また、オブジェクトの配置や入れ替え処理にはCommandパターン（`SwapBlockCommand`, `DeleteObjectCommand`など）を採用し、エディタに必須となる「Undo機能」を安全に実装しています。
**【今後の課題】**
オブジェクト生成時、ViewModel内で各コンポーネントへの依存性注入（DI）をif文の連続で行ってしまっているため、クラスが肥大化しています。今後はリフレクション機能の自作やイベントバスの導入により、ViewModelが個別のコンポーネント型を知らなくても自動でバインディングされるようなクリーンな設計を目指します。
